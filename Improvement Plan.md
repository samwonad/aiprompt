# Samwon Design Generator - Improvement Plan

## 1. 프로젝트가 여기까지 온 이유

처음 앱은 긴 프롬프트를 만들어주는 단일 HTML 도구에 가까웠다. 하지만 실제 사용에서 문제는 프롬프트 길이나 옵션 개수가 아니었다. 직원이 디자인 용어를 모르고, AI도 삼원에서 실제 납품/제작에 쓸 수 있는 결과 기준을 모른다는 것이 핵심 문제였다.

대화에서 정리된 결론은 다음과 같다.

- 프롬프트만 잘 만들면 해결된다는 접근은 실패 가능성이 높다.
- 옵션을 많이 넣으면 직원도 어렵고, 모델도 방향이 흔들린다.
- "인포그래픽형", "표지용", "깔끔하게" 같은 표현은 내부 제작 기준이 없으면 AI가 평균적으로 해석한다.
- 직원에게 프롬프트를 보여주고 수정하게 하면 결국 웹GPT와 다를 것이 없다.
- 필요한 것은 프롬프트 앱이 아니라, 삼원 내부의 제작 판단 기준을 담은 생성 시스템이다.
- 제작은 빠르게 하되, 실패하면 그 이유를 기록해서 다음 레시피 개선에 쓰는 구조가 필요하다.

그래서 기존 앱은 버리고, 새 프로젝트는 아래 흐름을 목표로 다시 시작했다.

```text
짧은 직원 입력
-> 앱이 제작 방향 3개 추천
-> 선택된 제작 레시피로 내부 프롬프트 생성
-> gpt-image-2로 최종 이미지 생성
-> 직원은 결과만 평가
-> 실패 이유와 프롬프트/레시피/옵션을 내부 기록으로 저장
-> 반복 실패를 보고 레시피 개선
```

## 2. 제품 원칙

### 직원 화면 원칙

- 직원은 긴 프롬프트를 보지 않아도 된다.
- 직원은 디자인 전문 용어를 몰라도 된다.
- 직원은 "무엇을 만들지", "어디에 쓸지", "어떤 느낌인지" 정도만 입력한다.
- 결과가 실패하면 버튼으로 이유를 남기고, 필요할 때만 짧은 메모를 남긴다.

### 관리자/개선 원칙

- 실제 프롬프트는 숨기되 반드시 저장한다.
- 어떤 레시피가 어떤 실패를 많이 내는지 레시피별 통계로 볼 수 있어야 한다.
- 왜 실패했는지 더 명확히 보기 위해 피드백에 참조 이미지를 첨부할 수 있어야 한다.
- 피드백이 바로 시스템을 자동 변경하면 위험하므로, 초반에는 사람이 레시피를 검토하고 수정한다.
- 완성 이미지 템플릿을 많이 저장하지 않는다. 조합 가능한 레시피와 실패 데이터를 저장한다.

### 생성 원칙

- 최종 이미지는 `gpt-image-2`가 만든다.
- 투명 배경은 프롬프트 문장에만 맡기지 않고 API 옵션으로 지정한다.
- 참조 이미지는 기본적으로 피드백 기록용이다. 자동으로 AI 분석에 보내지 않으며, 관리자가 선택했을 때만 분석 요청을 보내 비용을 통제한다.
- 기본 API 옵션은 다음 방향을 유지한다.

```text
model: gpt-image-2
background: transparent
output_format: png
quality: high
size: 1536x1024
```

## 3. 현재 기술 구조

현재는 단일 HTML이 아니라 `Vite + React + TypeScript` 앱으로 재구성했다.

```text
src/domain      제작 판단, 레시피, 프롬프트 조립
src/components  화면 컴포넌트
src/services    API 호출, 피드백 저장
netlify         서버리스 이미지 생성 함수
```

API 키는 브라우저 코드에 넣지 않는다. 실제 이미지 생성은 `netlify/functions/generate-design.mjs`에서 처리한다.

## 4. 파일별 역할

### 루트 파일

- `package.json`
  - 프로젝트 이름, 실행 명령, 의존성 관리.
  - 주요 명령: `npm run dev`, `npm run build`, `npm run lint`, `npm run test:analyzer`, `npm run test:e2e`.

- `index.html`
  - Vite 앱의 HTML 진입점.
  - 일반적으로 제목, 메타, root 엘리먼트만 수정한다.

- `netlify.toml`
  - Netlify 배포 및 로컬 Functions 실행 설정.
  - 서버리스 함수 경로는 `netlify/functions`.

- `.env.example`
  - 필요한 환경변수 예시.
  - 실제 키는 `.env`에 넣고 커밋하지 않는다.

- `README.md`
  - 실행 방법과 현재 MVP 범위 요약.

- `Improvement Plan.md`
  - 이 문서. 앞으로 프로젝트 방향과 구조를 이해하기 위한 기준 문서.

### 앱 진입점

- `src/main.tsx`
  - React 앱을 `#root`에 마운트한다.
  - 거의 수정할 일이 없다.

- `src/App.tsx`
  - 전체 화면 상태를 연결하는 컨테이너.
  - 직원 입력, 추천 결과, 선택 레시피, 생성 결과, 피드백 기록 상태를 관리한다.
  - 큰 기능 추가 시 여기서 흐름을 연결하되, 실제 로직은 domain/service/component로 분리한다.

- `src/index.css`
  - 전체 HTML/body/root 기본 스타일.
  - 전역 폰트, 기본 배경 정도만 관리한다.

- `src/App.css`
  - 현재 전체 UI 스타일.
  - 컴포넌트가 더 커지면 나중에 CSS 모듈 또는 파일 분리를 검토한다.

### 도메인 로직

- `src/domain/types.ts`
  - 앱 전체에서 공유하는 타입 정의.
  - 레시피, 분석 결과, 브리프, API 옵션, 피드백 기록, 참조 이미지 구조가 여기에 있다.
  - 데이터 구조를 바꾸기 전에는 이 파일부터 확인한다.

- `src/domain/recipes.ts`
  - 제작 레시피와 피드백 버튼 목록.
  - 현재 기본 레시피:
    - 표지에 넣기 좋은 대표 소스
    - 여러 내용을 한눈에 정리한 설명형 소스
    - 친근한 캐릭터 중심 소스
    - 간판/현수막에 크게 보이는 소스
    - 배경 없는 단일 오브젝트 소스
    - 아이콘/픽토그램 세트
  - 레시피 개선의 1차 수정 지점이다.

- `src/domain/analyzer.ts`
  - 직원 입력을 보고 사용처, 느낌, 키워드를 추론한다.
  - 각 레시피에 점수를 부여해 상위 3개 방향을 추천한다.
  - "왜 이 방향이 추천됐는지" 근거도 만든다.
  - 추천이 이상하면 먼저 이 파일의 힌트/점수 규칙을 고친다.

- `src/domain/promptBuilder.ts`
  - 선택된 레시피와 입력 내용을 실제 이미지 생성 프롬프트로 조립한다.
  - 기본 이미지 API 옵션도 여기에서 만든다.
  - 최종 모델은 `gpt-image-2`로 설정되어 있다.
  - 결과가 반복적으로 다른 방향으로 나가면 이 파일보다 먼저 `recipes.ts`의 레시피 규칙을 확인한다.

### 서비스 로직

- `src/services/designApi.ts`
  - 프론트에서 서버리스 함수로 생성 요청을 보내는 클라이언트.
  - 함수가 없거나 API 키가 없으면 드래프트 모드 결과로 돌아온다.

- `src/services/feedbackStore.ts`
  - 피드백 기록을 브라우저 `localStorage`에 저장한다.
  - 현재는 최대 300건 보관.
  - 나중에 Supabase, SQLite, 서버 DB로 바꾸려면 이 파일을 우선 교체한다.

- `src/services/referenceImages.ts`
  - 업로드/붙여넣기된 이미지를 피드백 기록용 참조 이미지로 변환한다.
  - `localStorage` 용량을 줄이기 위해 이미지를 최대 크기로 축소하고 JPEG data URL로 저장한다.
  - 이 이미지는 자동으로 생성 API에 보내지 않는다.

### 화면 컴포넌트

- `src/components/TopBar.tsx`
  - 상단 앱 제목, 기록 내보내기, 기록 초기화 버튼.

- `src/components/RequestPanel.tsx`
  - 직원 입력 영역.
  - 짧은 요청, 사용처, 느낌 선택, 방향 추천 버튼.

- `src/components/SuggestionList.tsx`
  - 추천된 제작 방향 3개를 보여준다.
  - 직원은 디자인 용어 대신 사람이 읽을 수 있는 방향명만 선택한다.

- `src/components/BriefPanel.tsx`
  - 선택된 레시피로 만든 제작 브리프를 보여준다.
  - 내부 프롬프트는 버튼을 눌러 확인할 수 있다.
  - 직원용이라기보다 관리자 확인용에 가깝다.

- `src/components/ResultPanel.tsx`
  - 생성 결과 이미지 또는 드래프트 상태를 보여준다.
  - 이미지가 있으면 PNG 다운로드 버튼을 제공한다.

- `src/components/FeedbackPanel.tsx`
  - 결과 평가 버튼.
  - "쓸만함", "배경이 생김", "너무 복잡함", "요소가 빠짐" 등의 실패 이유를 기록한다.
  - 짧은 메모 입력도 받는다. 메모는 필수가 아니며, 버튼만으로 빠르게 기록할 수 있어야 한다.
  - 참조 이미지 업로드와 캡처 이미지 붙여넣기를 지원한다.

- `src/components/DataPanel.tsx`
  - 저장된 피드백 통계와 최근 기록을 보여준다.
  - 레시피별 실패 우선순위를 계산해서 어떤 레시피를 먼저 고칠지 보여준다.
  - 앞으로 관리자 개선 화면으로 확장할 수 있다.

### 서버리스 함수

- `netlify/functions/generate-design.mjs`
  - OpenAI 이미지 API 호출 담당.
  - `OPENAI_API_KEY`가 없으면 실제 생성하지 않고 드래프트 모드로 응답한다.
  - 실제 생성 시 `gpt-image-2` 기본값과 투명 PNG 옵션을 사용한다.
  - API 키는 이 함수에서만 사용해야 하며 브라우저 코드로 보내면 안 된다.

## 5. 앞으로 고칠 때 기준

### 추천 방향이 이상할 때

1. `src/domain/analyzer.ts` 확인
2. 키워드 추론이 잘못됐는지 확인
3. 레시피 점수 가중치가 잘못됐는지 확인
4. 필요하면 `src/domain/recipes.ts`의 `triggers`를 수정
5. `tests/analyzer-cases.ts`에 실패한 입력을 추가해서 같은 문제가 반복되지 않게 한다

### 화면 흐름이 깨졌을 때

1. `npm run test:e2e`를 먼저 실행한다.
2. 입력, 추천, 생성 요청, 피드백 저장, 통계 반영 중 어느 단계에서 깨졌는지 확인한다.
3. 화면 컴포넌트 수정 후 `npm run build`, `npm run lint`, `npm run test:e2e`를 다시 실행한다.

### 생성 결과 스타일이 이상할 때

1. 먼저 실패 기록을 본다.
2. 같은 레시피에서 같은 실패가 반복되는지 확인한다.
3. `src/domain/recipes.ts`의 `rules`와 `avoid`를 수정한다.
4. 그래도 안 되면 `src/domain/promptBuilder.ts`의 공통 프롬프트 구조를 수정한다.

### 직원 화면이 복잡할 때

1. `src/components/RequestPanel.tsx`
2. `src/components/SuggestionList.tsx`
3. `src/components/FeedbackPanel.tsx`

이 세 곳을 우선 줄인다. 직원 화면에는 프롬프트 편집 기능을 늘리지 않는다.

### 피드백 저장 방식을 바꿀 때

현재는 `localStorage`다. 여러 직원이 함께 쓰려면 서버 저장소가 필요하다.

자세한 Supabase 전환 계획은 `supabase plan.md`를 기준으로 한다.

교체 우선순위:

```text
localStorage
-> Supabase 또는 간단한 서버 DB
-> 관리자 통계 화면
-> 레시피 버전별 성공률 추적
```

수정 시작 파일은 `src/services/feedbackStore.ts`.

### 실제 이미지 생성이 실패할 때

1. `.env`에 `OPENAI_API_KEY`가 있는지 확인
2. `npx netlify dev`로 실행했는지 확인
3. `netlify/functions/generate-design.mjs`의 요청/응답 확인
4. 모델, 크기, 품질, 투명 배경 옵션 확인

## 6. 다음 개발 순서

### Phase 1 - 실패 데이터가 쌓이는 MVP

- 실제 직원 입력 30개 테스트
- 각 생성 요청에 피드백 남기기
- 실패 기록 JSON 내보내기
- 레시피별 반복 실패 확인
- 실패 버튼과 짧은 메모를 함께 저장
- 참조 이미지를 함께 저장
- 레시피별 실패 우선순위 확인

완료 기준:

```text
어떤 레시피가 어떤 실패를 많이 내는지 볼 수 있다.
```

### Phase 2 - 실제 gpt-image-2 생성 연결

- `.env`에 `OPENAI_API_KEY` 연결
- `npx netlify dev`로 함수 포함 실행
- 결과 이미지 다운로드 확인
- 투명 PNG가 실제로 나오는지 검사

완료 기준:

```text
앱 안에서 생성된 PNG를 받아 실제 편집/인쇄 작업에 얹어볼 수 있다.
```

### Phase 3 - 피드백 기반 레시피 개선

- 실패 유형 상위 5개 집계
- 레시피별 `rules`/`avoid` 수정
- 수정 전후 성공률 비교

완료 기준:

```text
같은 입력군에서 실패 유형이 줄어드는지 수치로 확인한다.
```

### Phase 4 - 저장소와 관리자 화면

- 피드백을 브라우저가 아니라 서버 DB에 저장
- 레시피별 성공률/실패율 표시
- 프롬프트, API 옵션, 이미지 결과를 관리자만 확인
- 레시피 수정안 작성 화면 추가

완료 기준:

```text
여러 직원의 사용 기록이 한 곳에 쌓이고, 관리자가 개선할 수 있다.
```

### Phase 5 - 품질 검증 자동화

- 투명 배경 여부 검사
- 글자/로고 포함 의심 검사
- 이미지가 너무 복잡한지 수동/반자동 체크
- 실패 유형에 따른 재생성 프롬프트 보정

완료 기준:

```text
직원이 실패 버튼을 누르기 전에 앱이 일부 실패를 먼저 감지한다.
```

## 7. 당장 해야 할 실험

처음부터 완벽한 앱을 만들려고 하지 말고, 아래 20~30개 입력으로 테스트한다.

```text
표지 디자인에 쓸 농업 소스. 농부, 채소밭, 농기계, 채소바구니
```

각 결과에 대해 반드시 남길 것:


이 기록이 쌓이면 그 다음부터는 감으로 고치는 것이 아니라, 실패 데이터로 레시피를 고칠 수 있다.

## 8. 중요한 결론

이 프로젝트의 목표는 AI가 한 번에 완벽한 이미지를 만드는 것이 아니다.

목표는 다음이다.

```text
직원은 짧게 입력한다.
앱은 삼원 기준에 가까운 제작 방향을 잡는다.
gpt-image-2가 이미지를 만든다.
실패는 버리지 않고 기록한다.
기록을 바탕으로 레시피를 계속 개선한다.
```

이 구조를 지켜야 웹GPT보다 나은 사내 제작 도구가 된다.
